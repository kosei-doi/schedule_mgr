# スケジュール管理Webアプリ - 詳細企画書

## 1. プロジェクト概要

### 1.1 目的
ユーザーが日々の予定を効率的に管理できる、シンプルで使いやすいWebアプリケーションを提供する。

### 1.2 ターゲットユーザー
- 個人での予定管理が必要な方
- インストール不要でブラウザですぐ使えるツールを求めている方
- シンプルで直感的なインターフェースを好む方
- 複数デバイスで予定を同期したい方

### 1.3 主要な価値提供
- 依存関係なし、ブラウザで即座に動作
- Firebaseクラウド保存でデータ消失リスク低減
- 複数デバイス間でのリアルタイム同期
- クリーンで使いやすいUI

## 3. データモデル詳細

### 3.1 Event オブジェクト
```javascript
{
  id: string,           // ユニークID（Firebaseが自動生成）
                        // 例: "-NzxYZ123abc"
  
  title: string,        // 予定のタイトル（最大100文字）
                        // 必須項目
  
  description: string,  // 詳細説明（最大500文字）
                        // 任意項目、デフォルト: ""
  
  startTime: string,    // 開始時刻（ISO 8601形式）
                        // 例: "2025-10-20T10:00"
                        // 必須項目
  
  endTime: string,      // 終了時刻（ISO 8601形式）
                        // 例: "2025-10-20T11:00"
                        // 必須項目、startTimeより後であること
  
  color: string,        // カラーコード
                        // 例: "#3b82f6"
                        // デフォルト: "#3b82f6" (青)
  
  userId: string,       // ユーザーID（Firebase Authから取得）
  
  createdAt: string,    // 作成日時（ISO 8601形式）
  
  updatedAt: string     // 更新日時（ISO 8601形式、更新時のみ）
}
```

### 3.3 データ検証ルール
- **title**: 1-100文字、空白のみ不可
- **startTime**: 有効な日時形式、必須
- **endTime**: 有効な日時形式、startTimeより後、必須
- **description**: 0-500文字
- **color**: 有効な16進数カラーコード（#RRGGBB）

## 4. 機能仕様詳細

### 4.1 予定の追加

**画面遷移:**
1. ユーザーが「+ 予定追加」ボタンをクリック
2. モーダルフォームが表示される
3. フォームに情報を入力
4. 「保存」ボタンをクリック
5. バリデーション実行
6. 成功時: Firebaseに保存、モーダルが閉じ、ビューが更新される
7. 失敗時: エラーメッセージを表示

**入力項目:**
- タイトル（必須、テキスト）
- 説明（任意、複数行テキスト）
- 開始時刻（必須、datetime-local）
- 終了時刻（必須、datetime-local）
- 色（必須、6色から選択、デフォルト: 青）

**バリデーション:**
- タイトルが空でないこと
- 開始時刻が入力されていること
- 終了時刻が入力されていること
- 終了時刻が開始時刻より後であること

**エラーメッセージ:**
- "タイトルを入力してください"
- "開始時刻を入力してください"
- "終了時刻を入力してください"
- "終了時刻は開始時刻より後にしてください"

**Firebase処理:**
```javascript
const eventsRef = ref(db, `events/${userId}`);
const newEventRef = push(eventsRef);
await set(newEventRef, newEvent);
```

### 4.2 予定の編集

**画面遷移:**
1. ユーザーが既存の予定をクリック
2. モーダルフォームが表示される（編集モード）
3. 既存の情報がフォームに表示される
4. ユーザーが情報を編集
5. 「保存」ボタンをクリック
6. バリデーション実行
7. 成功時: Firebaseで更新され、ビューが更新される

**表示:**
- モーダルタイトル: "予定を編集"
- 削除ボタンが表示される
- すべてのフィールドに既存の値が入る

**Firebase処理:**
```javascript
const eventRef = ref(db, `events/${userId}/${eventId}`);
await update(eventRef, updatedEvent);
```

### 4.3 予定の削除

**画面遷移:**
1. 編集モード時に「削除」ボタンをクリック
2. 確認ダイアログが表示される
3. ユーザーが「OK」を選択
4. Firebaseから削除され、ビューが更新される
5. ユーザーが「キャンセル」を選択
6. 何も変更されない

**確認メッセージ:**
- "この予定を削除してもよろしいですか？"

**Firebase処理:**
```javascript
const eventRef = ref(db, `events/${userId}/${eventId}`);
await remove(eventRef);
```

### 4.4 日次ビュー

**表示内容:**
- 選択された日付の全予定
- 0時〜23時の時間軸
- 予定を時間軸上に配置（開始時刻と終了時刻に基づく）

**予定の配置:**
- 1時間 = 60px の高さ
- 予定の上端 = 開始時刻に基づく位置
- 予定の高さ = (終了時刻 - 開始時刻) に基づく高さ
- 重なる予定は横に並べて表示

**表示される情報:**
- 予定のタイトル
- 開始時刻〜終了時刻
- 背景色（選択された色）

### 4.5 週次ビュー

**表示内容:**
- 選択された週の7日間（日曜日〜土曜日）
- 各日付のヘッダー（曜日、日付）
- 各日の予定リスト

**予定の表示:**
- 時系列順にソート
- タイトル、開始時刻を表示
- 色付きのバー

**週の計算:**
- 現在の日付から、その週の日曜日を計算
- 日曜日から6日後までを表示

### 4.6 日付ナビゲーション

**操作:**
- 「←」ボタン: 前日/前週に移動
- 「→」ボタン: 翌日/翌週に移動
- 「今日」ボタン: 今日の日付に戻る

**表示:**
- 日次ビュー: "2025年10月20日（月）"
- 週次ビュー: "2025年10月20日〜26日"

### 4.7 ビュー切り替え

**操作:**
- 「日次」ボタン: 日次ビューに切り替え
- 「週次」ボタン: 週次ビューに切り替え

**動作:**
- アクティブなボタンに視覚的フィードバック（背景色変更）
- ビューがスムーズに切り替わる
- 選択中の日付は維持される

### 4.8 データの永続化とリアルタイム同期

**Firebase連携:**
- **リアルタイムリスナー**: データ変更を自動的に検知
- **自動同期**: 複数デバイス間で即座に反映
- **オフライン対応**: Firebase SDKが自動的にキャッシュを管理

**保存タイミング:**
- 予定を追加した直後
- 予定を編集した直後
- 予定を削除した直後

**読み込みタイミング:**
- ページ読み込み時
- 認証成功後
- データ変更時（リアルタイムリスナー）

**エラーハンドリング:**
- Firebase接続エラー: アラート表示
- 認証エラー: コンソールにログ出力、ユーザーに通知
- データ読み込みエラー: 再試行または空の状態で表示

## 6. Firebase初期設定手順

### 6.1 Firebaseプロジェクトの作成

1. **Firebase Console にアクセス**: https://console.firebase.google.com/
2. **プロジェクト `study-mgr` を選択**（既存の場合）
3. **新規プロジェクトの場合**: 「プロジェクトを追加」をクリック

### 6.2 Realtime Database の設定

1. **左メニューから「Realtime Database」をクリック**
2. **「データベースを作成」をクリック**
3. **ロケーション**: us-central1 を選択
4. **セキュリティルール**: テストモードで開始
5. **「有効にする」をクリック**

### 6.3 Authentication の設定

1. **左メニューから「Authentication」をクリック**
2. **「始める」をクリック**
3. **「サインイン方法」タブをクリック**
4. **「匿名」をクリック**
5. **「有効にする」トグルをON**
6. **「保存」をクリック**

### 6.4 セキュリティルールの設定

**Realtime Database ルール:**
```json
{
  "rules": {
    "events": {
      "$userId": {
        ".read": "auth != null && auth.uid == $userId",
        ".write": "auth != null && auth.uid == $userId"
      }
    }
  }
}
```

**設定手順:**
1. Realtime Database → ルール タブ
2. 上記ルールを貼り付け
3. 「公開」をクリック

### 6.5 Webアプリの登録

1. **プロジェクト設定（歯車アイコン）をクリック**
2. **「全般」タブをスクロール**
3. **「アプリを追加」 → Webアイコンをクリック**
4. **アプリのニックネーム**: 「スケジュール管理アプリ」
5. **Firebase Hosting は設定しない**
6. **「アプリを登録」をクリック**
7. **設定情報が表示される**（上記のfirebaseConfigと同じ）

## 8. セキュリティ考慮事項

### 8.1 Firebase セキュリティ

**Realtime Database ルール:**
- ユーザーは自分のデータのみ読み書き可能
- 認証必須
- ユーザーIDによるデータ分離

**Authentication:**
- 匿名認証使用
- ユーザーIDの自動管理
- セッション永続化

## 13. コスト見積もり

### 13.1 Firebase 無料枠（Spark プラン）

**含まれるもの:**
- Realtime Database: 1GB ストレージ、10GB/月 転送
- Authentication: 無制限
- Hosting: 10GB ストレージ、360MB/日 転送

**想定利用:**
- ユーザー数: 100人
- 1人あたりイベント数: 50件
- 合計: 5,000イベント ≈ 5MB
- 月間データ転送: < 1GB

**結論:** 無料枠で十分運用可能

### 13.2 有料プラン（Blaze プラン）

**必要になる場合:**
- ユーザー数が1,000人以上
- データ転送が10GB/月を超える
- Cloud Functions を使用する場合

**コスト（従量課金）:**
- ストレージ: $5/GB
- 転送: $1/GB
